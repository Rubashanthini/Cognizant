# spring-learn

Cognizant Digital Nurture 5.0 — **Spring REST Hands-on** project.

A Spring Boot / Spring Core demonstration project covering XML bean
configuration, SLF4J logging, bean scopes (Singleton vs Prototype), and
loading collections of beans from XML.

Spring Core and Spring Boot skills: Maven project setup, XML bean

vs Prototype bean scopes, and loading a `List` of beans from XML.

## 3. Spring Boot Features Used

- `@SpringBootApplication` — auto-configuration, component scanning, and configuration in one annotation.
- `SpringApplication.run()` — bootstraps the embedded Tomcat server and Spring `ApplicationContext`.
- `CommandLineRunner` — executes exercise demos automatically right after startup.
- `application.properties` — centralized configuration for logging and server port.
- Spring Boot DevTools — automatic restart during development.
- Embedded Tomcat — no external application server required.

## 4. Spring Core Features Used

- **XML Bean Configuration** (`date-format.xml`, `country.xml`) — beans defined declaratively rather than via annotations.
- **`ClassPathXmlApplicationContext`** — loads a standalone Spring container from an XML resource on the classpath.
- **Constructor Injection** (`<constructor-arg>`) — used to populate the `Country` bean's fields.
- **Bean Scopes** — `singleton` (default) and `prototype`, demonstrated side by side.
- **Collections support** (`<list>`, `<ref>`) — used to assemble a `java.util.ArrayList` of `Country` beans.

## 5. How to Import into Eclipse

1. Open Eclipse and go to **File → Import…**
2. Select **Maven → Existing Maven Projects** and click **Next**.
3. Click **Browse…**, select the `spring-learn` root folder (the one containing `pom.xml`), and click **Finish**.
4. Eclipse will download dependencies and build the project automatically.
5. Right-click the project → **Run As → Spring Boot App** (or **Java Application**, targeting `SpringLearnApplication`).

## 6. How to Import into IntelliJ IDEA

1. Open IntelliJ IDEA and choose **File → Open…**
2. Select the `spring-learn` folder (containing `pom.xml`) and click **OK**.
3. IntelliJ will detect it as a Maven project and prompt to import — accept it, or right-click `pom.xml` → **Add as Maven Project**.
4. Wait for Maven dependencies to resolve.
5. Locate `SpringLearnApplication.java` and click the green **Run** icon next to the `main` method.

## 7. How to Import into VS Code

1. Install the **Extension Pack for Java** and **Spring Boot Extension Pack** from the VS Code marketplace.
2. Open VS Code and choose **File → Open Folder…**, then select the `spring-learn` directory.
3. VS Code will detect `pom.xml` and automatically configure the Java project.
4. Open `SpringLearnApplication.java`, and click **Run** (or use the Spring Boot Dashboard panel) to start the application.

## 11. Singleton vs Prototype — Explanation

| Aspect                  | Singleton (default)                                   | Prototype                                             |
|--------------------------|---------------------------------------------------------|----------------------------------------------------------|
| Instances created         | Exactly **one** per Spring container                    | **New instance** every time `getBean()` is called        |
| `scope` attribute         | `scope="singleton"` (or omitted — it's the default)      | `scope="prototype"`                                        |
| Constructor invocations   | Once                                                     | Once per `getBean()` call                                  |
| Object identity (`==`)    | All references point to the **same** object              | Each retrieved reference is a **distinct** object           |
| Typical use case          | Stateless services, shared configuration, shared resources | Stateful objects, objects that hold per-use / per-request data |
| Container lifecycle mgmt  | Fully managed (created, cached, destroyed by container)  | Container creates and hands off the instance, but does **not** manage its destruction |

`BeanScopeService` in this project retrieves both a `singletonCountry` bean
and a `prototypeCountry` bean multiple times and compares their identity
hash codes (`System.identityHashCode`) to make this difference directly
observable in the console output.

## 12. XML Bean Configuration — Explanation

Spring supports defining beans declaratively in XML files instead of (or in
addition to) using annotations such as `@Component` or `@Bean`. An XML bean
definition file:

- Is wrapped in a root `<beans>` element referencing the Spring beans XSD schema.
- Contains one `<bean>` element per managed object, each with a unique `id` and a fully-qualified `class`.
- Can inject dependencies via `<constructor-arg>` (constructor injection) or `<property>` (setter injection).
- Can define collections such as `<list>`, `<set>`, `<map>`, and `<props>`, and reference other beans using `<ref bean="..."/>`.
- Can control a bean's lifecycle scope with the `scope` attribute (`singleton` by default, or `prototype`, `request`, `session`, etc.).

This project uses two such XML files:

- **`date-format.xml`** — defines a single `java.text.SimpleDateFormat` bean configured with the pattern `dd/MM/yyyy`.
- **`country.xml`** — defines `Country` beans (singleton and prototype scoped), four individual country beans, and an `ArrayList` bean (`countryList`) that aggregates references to those four beans via `<list>` and `<ref>`.

Both files are loaded independently of the Spring Boot auto-configured
`ApplicationContext` using `new ClassPathXmlApplicationContext("<file>.xml")`,
exactly as required by Exercises 2, 4, 5 and 6, and beans are retrieved
using `context.getBean(...)`.

## 13. Logging

All logging is implemented using **SLF4J** (`org.slf4j.Logger` /
`LoggerFactory`) — `System.out.println()` is never used anywhere in this
project. Every method logs a `START` message on entry and an `END` message
on exit, with `DEBUG` level logs added inside constructors, getters and
setters of `Country`, and `ERROR` level logs used for exception handling
(see `DateFormatService.displayDate()`).

Logging levels and the console pattern are configured centrally in
`application.properties`:

```properties
logging.level.root=INFO
logging.level.org.springframework=INFO
logging.level.com.cognizant.springlearn=DEBUG
logging.pattern.console=%d{yyyy-MM-dd HH:mm:ss} - %-5level [%thread] %logger{36} - %msg%n
```
